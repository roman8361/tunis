# Техническое задание (v2)

## Веб-сервис «Тунисский формат»
### Формат: Тунисский формат для 5 игроков

---

## 1. Цель проекта
Разработать веб-сервис «Тунисский формат» для проведения мини-турниров по пляжному волейболу в тунисском формате для 5 игроков, где в каждом туре участвуют 4 игрока, а 1 игрок пропускает тур.

Сервис должен позволять:
- регистрироваться и входить в систему по email и паролю;
- создавать турниры;
- видеть только свои турниры для обычного пользователя;
- проводить турнир из 15 туров;
- автоматически формировать туры;
- сохранять результаты игр;
- редактировать уже сыгранные туры;
- вручную корректировать пары;
- автоматически считать статистику и итоговые места;
- сохранять историю турниров;
- иметь административную панель для суперпользователя.

Основная задача приложения — **удобно вести счёт турнира, автоматически определять итоговые места и хранить турниры в личном кабинете пользователя**.

---

## 2. Язык интерфейса
Весь интерфейс приложения должен быть полностью на русском языке.

Это требование распространяется на:
- кнопки;
- поля ввода и подписи;
- ошибки валидации;
- уведомления;
- таблицы;
- статусы;
- служебные сообщения;
- навигацию;

- системные сообщения, которые видит пользователь.

Использование английского языка в пользовательском интерфейсе не допускается, кроме технически необходимых внутренних значений в коде, базе данных и API.

## 3. Общая логика турнира

### 3.1 Участники
- В одном турнире всегда участвуют ровно **5 игроков**.
- Пользователь вручную вводит имена всех 5 игроков перед стартом турнира.
- Игроки внутри турнира не являются отдельными аккаунтами системы, а представляют участников конкретного турнира.

### 3.2 Формат одного тура
- В каждом туре играют **4 игрока**.
- **1 игрок пропускает тур**.
- Игра проходит в формате **2 на 2**.
- Каждый игрок играет **только за себя**, то есть в итоговой таблице учитываются его личные результаты независимо от того, с кем он был в паре.

### 3.3 Количество туров
- Всего в одном турнире: **15 туров**.

### 3.4 Правила ротации
Необходимо автоматически генерировать туры по следующим правилам:

1. После каждого тура игрок, который пропускал предыдущий тур, **обязательно участвует** в следующем туре.
2. В каждом туре **один игрок пропускает тур**.
3. Все игроки должны пропускать туры **максимально равномерно**.
4. Нужно **максимально перемешивать пары** между турами.
5. Желательно минимизировать повторение:
   - одинаковых пар в одной команде;
   - одинаковых соперников;
   - одинаковых составов тура.

### 3.5 Распределение пропусков
Так как туров 15, а игроков 5, суммарно будет 15 пропусков. Значит каждый игрок должен пропускать **ровно 3 тура**.

Это обязательное условие логики турнира, но **эту информацию не нужно выводить в интерфейсе и итоговых таблицах**.

## 4. Правила счёта

### 4.1 Выбор длины партии
Перед стартом турнира пользователь выбирает один из вариантов:
- игра до **11 очков**;
- игра до **15 очков**;
- игра до **21 очка**.

### 4.2 Ввод счёта
После каждого тура пользователь вручную вводит итоговый счёт партии.

Примеры:
- 11:8
- 15:13
- 21:19

### 4.3 Валидация счёта
Приложение должно проверять корректность введённого результата:

1. Одна из команд обязана первой набрать ровно выбранный лимит очков: 11, 15 или 21.
2. Вторая команда должна набрать меньше.
3. Победа может быть с **любой разницей очков**, в том числе в 1 очко.
4. Нельзя вводить ничью.
5. Нельзя вводить отрицательные значения.
6. Нельзя завершить тур без ввода счёта.

### 4.4 Начисление статистики по туру
После завершения тура для каждого из 4 игравших игроков обновляется статистика:

- игрокам победившей пары:
  - +1 выигранная партия;
  - +разница очков в партии;
- игрокам проигравшей пары:
  - +0 выигранных партий;
  - −разница очков в партии;
- игроку, который не участвовал в данном туре:
  - статистика за тур не начисляется.

### 4.5 Правило подсчёта очков
Очки игрока считаются не как сумма набранных командой мячей, а как **разница очков по итогам партии**.

Пример:
- счёт тура: **11:9**
- разница: **2**
- оба игрока команды-победителя получают **+2 очка**
- оба игрока проигравшей команды получают **−2 очка**

Ещё пример:
- счёт тура: **15:14**
- разница: **1**
- победители получают **+1 очко** каждый
- проигравшие получают **−1 очко** каждый

## 5. Определение мест

После завершения всех 15 туров приложение должно определить итоговые места игроков.

### 5.1 Главные критерии сортировки
Игроки сортируются по следующим правилам:

1. **Сначала по количеству выигранных партий** — больше выше.
2. **При равенстве выигранных партий — по количеству очков по разнице мячей** — больше выше.

### 5.2 Отображаемые показатели в итоговой таблице
Для каждого игрока вывести:
- место;
- имя;
- сыграно туров;
- выигранные партии;
- проигранные партии;
- всего очков по разнице мячей.

### 5.3 Формулы
- **Сыграно туров** = количество туров, в которых игрок участвовал
- **Проигранные партии** = сыграно туров − выигранные партии
- **Итоговое место** определяется сортировкой по правилам из пункта 5.1

## 6. Пользователи, роли и доступ

### 6.1 Типы пользователей
В системе предусмотрены две роли:
- `user` — обычный пользователь;
- `superadmin` — суперпользователь / администратор.

### 6.2 Регистрация обычного пользователя
Пользователь может зарегистрировать аккаунт по:
- email;
- паролю.

Требования:
- email обязателен;
- email должен быть уникальным;
- пароль обязателен;
- пароль хранится только в виде хэша;
- пользователь после регистрации получает роль `user`.

### 6.3 Вход в систему
Пользователь входит в систему по:
- email;
- паролю.

При успешном входе создаётся авторизованная сессия или выдаётся токен доступа.

### 6.4 Системный суперпользователь
В системе должен существовать преднастроенный суперпользователь:
- email: **admin@local**
- пароль: **admin**
- роль: **superadmin**

Требования:
- суперпользователь создаётся автоматически при первом запуске системы, если ещё не существует;
- `admin@local` является системной учётной записью и не обязан быть реальным email;
- пароль `admin` допустим как стартовый системный пароль для v2, но в продакшене должен рассматриваться как небезопасный.

### 6.5 Ограничение доступа для обычного пользователя
Обычный пользователь:
- видит только свои турниры;
- может создавать, редактировать и удалять только свои турниры;
- не видит других пользователей;
- не имеет доступа к административной панели.

### 6.6 Права суперпользователя
Суперпользователь может:
- видеть список всех пользователей сервиса;
- просматривать данные каждого пользователя;
- удалять любого пользователя;
- просматривать все турниры любого пользователя;
- просматривать результаты игр любого пользователя;
- редактировать любые турниры любого пользователя;
- удалять любые турниры любого пользователя.

### 6.7 Правило удаления пользователя
При удалении пользователя необходимо удалять **все связанные с ним данные**:
- аккаунт пользователя;
- все его турниры;
- все раунды и результаты этих турниров;
- связанные служебные записи, если они есть.

Удаление должно происходить **полностью, без возможности восстановления**, если отдельно не предусмотрен механизм soft delete.

## 8. Сводная таблица результатов

Итоговая таблица должна быть наглядной и включать все расчёты.

### Предлагаемые колонки
| Место | Игрок | Сыграно | Побед | Поражений | Очки |
|------|------|--------:|------:|----------:|-----:|

### Правила вычисления
- **Сыграно** = число туров, где игрок принимал участие
- **Побед** = число туров, где игрок входил в победившую пару
- **Поражений** = сыграно − побед
- **Очки** = сумма разниц очков игрока по всем сыгранным турам
- Для победы игрок получает положительную разницу очков
- Для поражения игрок получает отрицательную разницу очков

## 9. Алгоритм генерации туров

### 9.1 Требования к алгоритму
Алгоритм должен до старта турнира сгенерировать список из 15 туров.

Для каждого тура нужно определить:
- одного пропускающего игрока;
- 4 играющих игроков;
- разбивку этих 4 игроков на 2 пары.

### 9.2 Ограничения
Алгоритм должен стремиться соблюдать:
1. каждый игрок пропускает ровно 3 тура;
2. игрок, пропускавший прошлый тур, не может пропускать следующий тур;
3. пары должны повторяться как можно реже;
4. сочетания соперников должны быть как можно разнообразнее.

### 9.3 Практическая реализация
Рекомендуется использовать один из подходов:

#### Вариант A — предгенерация с оптимизацией
1. Сначала создать последовательность пропускающих игроков на 15 туров так, чтобы:
   - каждый игрок встречался ровно 3 раза;
   - не было двух одинаковых пропусков подряд.
2. Для каждого тура подбирать пары среди 4 активных игроков.
3. Оценивать качество раскладки по штрафной функции:
   - штраф за повтор одной и той же пары;
   - штраф за частое повторение тех же соперников;
   - штраф за одинаковые составы.
4. Выбирать расклад с минимальным штрафом.

#### Вариант B — жадный алгоритм
На каждом туре:
- фиксировать пропускающего;
- среди возможных разбиений 4 игроков на 2 пары выбирать то, которое меньше всего повторяет предыдущие пары и соперничества.

### Рекомендация
Для v2 допустим **жадный алгоритм**, но код должен быть написан так, чтобы позже можно было заменить его на более сильный оптимизационный алгоритм.

## 12. Нефункциональные требования

- Веб-приложение для запуска в браузере.
- Оптимизация в первую очередь под мобильные устройства, так как счёт вероятнее всего будут вести с телефона на площадке.

### 12.2 Производительность
- Приложение должно работать быстро без заметных задержек.
- Турнирная логика должна пересчитываться без ощутимых лагов.
- Допускается серверная обработка данных при сохранении и загрузке турниров.

### 12.3 Надёжность
- Нельзя терять введённые данные при случайном обновлении страницы.
- Текущее состояние активного турнира должно сохраняться на сервере.
- История турниров должна храниться в базе данных.
- Доступ к данным должен зависеть от роли и владельца данных.

### 12.4 Безопасность
- Пароли не должны храниться в открытом виде.
- Пароли должны храниться только как хэш.
- Защита маршрутов обязательна.
- Административные маршруты доступны только роли `superadmin`.
- Пользовательские маршруты должны проверять право доступа к турниру.
- Сброс пароля в версии v2 **не требуется**.

### 12.5 Простота использования
- Интерфейс должен быть понятным без обучения.
- Основной сценарий: войти — создать турнир — ввести игроков — начать — после каждого тура занести счёт — получить итог.

## 16. API и серверная логика

### 16.1 Auth API
- `POST /auth/register` — регистрация
- `POST /auth/login` — вход
- `POST /auth/logout` — выход
- `GET /auth/me` — получение текущего пользователя

### 16.2 User Tournament API
- `GET /tournaments` — получить список своих турниров
- `POST /tournaments` — создать турнир
- `GET /tournaments/:id` — получить один свой турнир
- `PUT /tournaments/:id` — обновить турнир
- `DELETE /tournaments/:id` — удалить свой турнир

### 16.3 Admin API
- `GET /admin/users` — список пользователей
- `GET /admin/users/:id` — данные пользователя
- `DELETE /admin/users/:id` — удалить пользователя вместе со всеми данными
- `GET /admin/users/:id/tournaments` — турниры пользователя
- `GET /admin/tournaments/:id` — получить любой турнир
- `PUT /admin/tournaments/:id` — обновить любой турнир
- `DELETE /admin/tournaments/:id` — удалить любой турнир

### 16.4 Требования к API
- все защищённые маршруты требуют авторизации;
- обычный пользователь не может получить доступ к чужим турнирам;
- маршруты `/admin/*` доступны только роли `superadmin`.

## 17. Обработка ошибок и edge cases

Приложение должно учитывать:
1. Повторяющиеся имена игроков — запрещать старт.
2. Пустые имена — запрещать старт.
3. Невалидный счёт — показывать понятную ошибку.
4. При редактировании уже сыгранного тура статистика должна полностью и корректно пересчитываться.
5. При ручной корректировке пар нельзя нарушать правило: 4 активных игрока, 1 пропускает, 2 на 2.
6. Попытку перейти дальше без сохранения результата — запрещать.
7. Обновление страницы — восстанавливать текущий турнир с сервера.
8. История турниров должна сохраняться между сессиями.
9. Кнопку сброса турнира — запрашивать подтверждение.
10. Попытку регистрации с уже занятым email — отклонять.
11. Попытку входа с неверным email или паролем — показывать как ошибку авторизации.
12. Попытку доступа к чужому турниру — запрещать.
13. Попытку доступа обычного пользователя к админке — запрещать.
14. При удалении пользователя каскадно удалять все его турниры и результаты.

## 21. Краткая формулировка задачи

Нужно создать адаптивное веб-приложение «Тунисский формат» на React + TypeScript + Tailwind + Node.js + Express + PostgreSQL для ведения турнира по пляжному волейболу в тунисском формате для 5 игроков. Пользователь регистрируется и входит в систему по email и паролю, после чего попадает в личный кабинет и видит только свои турниры. В приложении пользователь выбирает лимит очков (11/15/21), вводит 5 имён, после чего система автоматически генерирует 15 туров. В каждом туре играют 4 человека (2 на 2), 1 игрок пропускает тур, все пропуски распределяются поровну, а игрок, пропускавший предыдущий тур, обязательно играет в следующем. Пользователь должен иметь возможность заранее просмотреть все 15 туров, вручную корректировать пары и редактировать результаты уже сыгранных туров. После каждого тура пользователь вводит итоговый счёт, а приложение пересчитывает статистику игроков. Места определяются сначала по числу выигранных партий, затем по очкам, где очки считаются как суммарная разница мячей игрока по всем турам. В системе должен существовать суперпользователь `admin@local / admin`, который может видеть всех пользователей, удалять их вместе со всеми данными, а также просматривать, редактировать и удалять любые турниры. Интерфейс должен быть в пляжной стилистике, полностью на русском языке, с современными читаемыми шрифтами без пиксельных шрифтов, с большим количеством анимаций, хорошей адаптацией под мобильные устройства, страницами «О сервисе» и «О команде» и серверным хранением данных в базе.
